# 评估 Agent 设计说明

## 目标

`Evaluator Agent` 负责**对候选人表现进行结构化评估**：根据题目与对话内容提取证据、判断覆盖面与深度、给出评分/等级/风险点/改进建议，并产出可落盘的评估结果。

## 位置与依赖

- **文件**: `agents/evaluator.py`
- **依赖**
  - `services/llm.py`：通过 `LLMClient.chat/achat` 调用模型
  - `config/prompts.yaml`：评估提示词模板（评分维度、证据要求、输出格式）
  - `storage/evaluations/`：评估结果存储（后续实现）
  - `agents/supervisor.py`：由 `Supervisor` 状态机在 `EVALUATE` 阶段触发

## 职责边界

- **负责**
  - 依据 rubric 打分（维度化）
  - 强制“证据驱动”（每个结论对应对话证据）
  - 识别风险（背诵、概念混淆、关键缺失、工程经验不足）
  - 输出结构化评估结果（便于 UI 展示与持久化）
- **不负责**
  - 生成面试问题（交给 `Interviewer Agent`）
  - 控制对话节奏（交给 `Supervisor`/上层 orchestrator）
  - 最终录用决策（可由上层业务规则综合判断）

## 输入与输出（建议数据结构）

### 输入

- 当前题目与上下文
  - `question`: 题干、期望能力点、标签、难度
  - `candidate_answer`: 候选人主回答
  - `followups`: 追问与回答列表
  - `round_summary`: 面试官提供的结构化摘要（如有）
- 面试设置
  - 级别/岗位
  - 评分区间或等级定义

### 输出（结构化评估结果）

建议输出字段：

- `overall_score`：总体分（例如 0-10 或 0-100）
- `dimension_scores`：维度分（概念理解、系统设计、编码能力、排错能力、工程意识等）
- `evidence`：证据列表（引用候选人原话或摘要片段）
- `strengths`：亮点
- `gaps`：缺口
- `risk_flags`：风险标签（如“回答含糊”“关键概念错误”“无法落地”）
- `recommended_followups`：建议下轮追问方向（可选）
- `decision_hint`：仅提示（如 strong_yes / yes / no / strong_no），最终决策仍在上层

## 评分 Rubric（建议在 prompts 中固化）

常见维度示例：

- **概念准确性**：术语/原理是否正确
- **解释深度**：能否解释“为什么”和“怎么做”
- **边界与反例**：能否说明限制、失败场景、权衡
- **工程落地**：能否给出实践方案、工具链、监控与回滚
- **沟通结构**：表达是否有条理、能否澄清问题

强调规则：

- **不确定则标注不确定**：不要编造候选人未提到的能力
- **结论必须有证据**：每条评价对应对话片段或明确推断链

## 与 LLM 的交互约定

- `system`：评估官身份 + rubric + 输出格式（强制结构化）
- `user`：提供题目、对话记录、追问记录、面试官摘要

建议输出格式为可解析结构（JSON/严格字段分段）。若输出不可解析，采取：

- 降低温度重试一次
- 强化格式约束（例如“只输出 JSON，不要额外文本”）

## 可配置项（建议来自 `config/settings.yaml`）

- 评分尺度（0-5 / 0-10 / 0-100）
- 维度集合与权重
- 证据条目最小数量
- 输出语言（中文/英文）

## 容错与一致性策略

- 同一候选人在多题评估时：
  - 保持评分尺度一致
  - 使用统一风险标签集合
- 当输入对话不足以判断：
  - 明确标注“信息不足”
  - 提供可操作的后续追问建议

## 扩展点

- 题目级评估 → 会话级综合评估（跨题聚合）
- 引入结构化日志与可解释性（证据链）
- 与题库能力矩阵联动，动态调整后续选题
- 输出到 `storage/evaluations/` 并生成可视化报告（UI）
